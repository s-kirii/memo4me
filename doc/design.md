# memo4me 設計書

## 1. 目的

`memo4me` は、Notion ライクな操作感を持ちながら、よりシンプルに使える個人用メモアプリを目指す。

本アプリの主目的は以下。

- Markdown 形式で気軽にメモを書く
- メモをタグや更新日などで整理する
- ローカル環境だけで安全かつ簡単に動かす
- GitHub から取得して、複数 PC に比較的簡単に導入する

初期バージョンでは、外部サーバー、外部 DB、認証基盤、共同編集は扱わない。

## 2. 想定ユーザーと前提

- 想定ユーザーは作者本人 1 名
- 利用端末は PC 中心
- まずはローカル完結で利用する
- 社用 PC を含む複数端末で導入しやすい構成を重視する
- OS や実行環境の差異はなるべくアプリ側で吸収する

このため、Web サービスではなく `ローカル Web アプリ` として設計する。

## 3. 設計方針

### 3.1 方針の要約

- UI は `React` で構築する
- フロントエンド開発とビルドには `Vite` を使う
- エディタには `Tiptap` を採用し、Notion ライクな編集体験を重視する
- データ操作はローカル API サーバーが担当する
- データはローカルの `SQLite` に保存する
- 起動はワンクリックで行い、ブラウザを自動で開く
- ネットワーク公開はせず、`127.0.0.1` のみに bind する
- 初期実装は `Node.js` を採用し、将来 `Go` に移行しやすい疎結合構成にする

### 3.2 この構成を選ぶ理由

- ブラウザ UI は慣れていて開発しやすい
- React により Notion ライクな編集画面や一覧画面を組みやすい
- Tiptap により、表示と編集が自然につながるリッチな体験を実現しやすい
- SQLite はローカル保存と検索に強く、JSON や localStorage より拡張しやすい
- ローカル API を分けることで、UI とデータ管理の責務を分離できる
- 将来的に同期や機能追加をする場合も構造を崩しにくい
- API 境界を明確にすることで、将来 Node.js バックエンドを Go に置き換えやすくする

## 4. スコープ

### 4.1 現在のアプリ本体に含める

- メモの作成、閲覧、編集、削除
- タイトル
- Markdown ベースの本文編集
- Notion ライクなインライン編集体験
- タグ付け
- 作成日時、更新日時
- メモ一覧表示
- タイトル、本文、タグを対象とした検索
- タグによる絞り込み
- 更新日順などの並び替え
- ローカル保存
- 開発用ワンコマンド起動
- 将来のワンクリック起動とブラウザ自動起動
- AI 設定モーダル
- AI アシスタントによる要約、構造化、タスク抽出、自由入力
- `メモ` / `タスク` のワークスペース切り替え
- テーマ切り替え

### 4.2 現在もスコープ外とするもの

- 外部サーバー
- 外部 DB
- ログイン、認証、権限管理
- 複数ユーザー
- 共同編集
- PC 間自動同期
- Notion のような自由な block editor
- 複雑な DB リレーション
- モバイル最適化最優先

補足:

- AI 機能とタスクリストはすでに実装済み
- ただし複数ユーザー、外部公開、共同編集は別 PJ として扱う

## 5. 画面構成

現状は、操作性を優先したシンプルな 2 カラム構成を採用している。

### 5.1 画面一覧

- メイン画面
- 新規メモ作成
- メモ詳細編集

現状実装は `メイン画面 + 右ペイン編集` の単一画面構成とする。

### 5.2 レイアウト案

```text
+------------------------------------------------------------------+
| Header: memo4me / [メモ] [タスク] / 新規メモ / 設定 / 終了      |
+----------------------------+-------------------------------------+
| Sidebar                    | Main                                |
| - 絞り込み                 | - 更新日時 / 保存状態               |
|   - 検索                   | - タイトル入力                      |
|   - 並び順                 | - タグ編集                          |
|   - 関連タスク導線         | - 関連タスク                        |
|   - タグフィルタ           | - 本文区切り                        |
| - メモ一覧                 | - AIアシスタント                    |
|   - タイトル               | - リッチテキストエディタ            |
|   - 抜粋(2行程度)          |                                     |
|   - 更新日時 / タグ        |                                     |
+----------------------------+-------------------------------------+
```

### 5.3 UI の基本ルール

- 左側に一覧とフィルタ
- 右側に編集中のメモ
- Header には `メモ` / `タスク` のワークスペースタブ、`新規メモ`、設定メニュー、終了ボタンを置く
- 検索は左カラムの絞り込み領域に置く
- 編集中の内容は自動保存を前提とする
- メモ画面には `関連タスク` と `このメモのタスクを管理` 導線を置き、タスク画面へ自然に移動できるようにする
- タスク画面は `左: ダッシュボード / 右: 管理キャンバス` とし、ダッシュボードには工数ベースの炎上予報を表示する
- タスクは `想定工数[H]` と `進捗率[%]` を持ち、`0%=未着手 / 10-90%=進行中 / 100%=完了` として扱う
- 表示と編集を明確に分断しすぎず、その場で自然に編集へ入れる体験を重視する
- Enter や Markdown 記法に応じて見た目が整う、Notion ライクな使用感を目指す
- 初期 UI は Notion 完全再現ではなく、メモ編集体験と管理体験に集中する
- 左右カラムは独立スクロールとし、左側は `Filters` を固定する

## 6. システム構成

### 6.1 全体構成

```text
[ユーザー]
   ↓
[Google Chrome]
   ↓ http://127.0.0.1:{port}
[ローカルアプリ本体]
   ├─ 静的ファイル配信
   ├─ ローカル API
   ├─ ブラウザ自動起動
   └─ SQLite 操作
          ↓
      [SQLite DB]
```

### 6.2 開発時構成

```text
[React + Vite dev server]
   └─ UI 表示

[Node.js Local API server]
   ├─ CRUD API
   └─ SQLite 操作
```

開発時はフロントエンドと API サーバーを別プロセスで動かす。現状実装ではフロントエンドからローカル API を直接呼ぶ。

現状はルートの `dev.sh` から、frontend と backend をまとめて起動して Chrome を開ける。

### 6.3 配布時構成

```text
[launcher]
   ├─ 空きポートを確保
   ├─ ローカル API サーバーを起動
   ├─ ビルド済みフロントを配信
   └─ ブラウザで localhost を開く
```

配布後は Vite を使わない。ビルド済みの静的ファイルをローカルサーバーから配信する。

## 7. 技術スタック案

### 7.1 フロントエンド

- React
- Vite
- TypeScript
- Tiptap
- React 標準 state

候補:

- エディタ: `@tiptap/react`
- 拡張: `StarterKit`、`Placeholder`、`CodeBlockLowlight`
- Markdown 変換: 独自変換レイヤ
- データ取得: `fetch + useEffect`
- スタイリング: CSS Modules または Tailwind CSS

### 7.2 ローカルバックエンド

候補は 2 つある。

#### A. Node.js / TypeScript

メリット:

- React 経験と近く学習しやすい
- JavaScript/TypeScript で統一しやすい
- 社用 PC に npm が入っているため初期導入の障壁が低い
- まずプロトタイプを速く作り、編集体験の検証を進めやすい

デメリット:

- 配布時にランタイム同梱や単体実行ファイル化がやや面倒
- 環境差分吸収は Go より重くなりやすい

#### B. Go

メリット:

- 単体バイナリ化しやすい
- Windows/macOS 対応しやすい
- 起動ランチャーと HTTP サーバーをまとめやすい

デメリット:

- 学習コストが少しある

#### 推奨

初期実装は `Node.js + React/Vite + SQLite` を採用する。  
理由は、開発者の習熟度が高く、社用 PC 側にも npm があり、最優先である編集体験の試作と改善を進めやすいため。

配布改善の本線は、当面 `Electron + installer` とする。  
これにより、ユーザーに `Node.js` / `npm` / 外部ブラウザを意識させずに `.app` / `.exe` として配布しやすくする。

一方で、将来 Web アプリとして公開する可能性も残す。  
そのため、Electron は薄い shell に留め、初期実装段階から API 契約を固定し、フロントエンドとバックエンドを疎結合に保つ。

`Go` への移行は当面の優先事項ではなく、将来の配布要件や性能要件が明確になった時点で再評価する。

### 7.3 DB

- SQLite

理由:

- 外部インストール不要
- 単一ファイルで扱いやすい
- 検索、並び替え、将来の拡張に強い

## 9. データモデル設計

MVP では「メモ本文」と「管理用メタデータ」を分けて考える。

### 9.1 エンティティ

- notes
- tags
- note_tags

### 9.2 テーブル定義案

#### notes

| column | type | required | note |
|---|---|---|---|
| id | text | yes | UUID |
| title | text | yes | メモタイトル |
| content_md | text | yes | Markdown 本文 |
| content_plain | text | no | 検索用プレーンテキスト |
| is_archived | integer | yes | 0 or 1 |
| created_at | text | yes | ISO 8601 |
| updated_at | text | yes | ISO 8601 |

#### tags

| column | type | required | note |
|---|---|---|---|
| id | text | yes | UUID |
| name | text | yes | 表示名 |
| normalized_name | text | yes | 正規化名 |
| created_at | text | yes | ISO 8601 |

#### note_tags

| column | type | required | note |
|---|---|---|---|
| note_id | text | yes | FK |
| tag_id | text | yes | FK |

### 9.3 インデックス

- `notes(updated_at desc)`
- `notes(title)`
- `tags(normalized_name)`
- `note_tags(note_id, tag_id)`

検索性能を上げたくなったら、後で SQLite FTS を検討する。

### 9.4 設計上のポイント

- `content_md` を永続化の基準フォーマットとして保持する
- 検索しやすくするため `content_plain` を派生保持してもよい
- タグは多対多で設計する
- タグは表示名と正規化名を分けて扱う
- 初期はカスタムプロパティを入れない
- エディタ内部表現と保存形式を分離し、将来的なバックエンド移行や変換方式の変更に耐えられるようにする

## 10. API 設計

ローカルアプリであっても、UI とデータ層を分けるため HTTP API を設ける。

### 10.1 API 一覧

#### health

- `GET /api/health`

レスポンス例:

```json
{
  "status": "ok"
}
```

#### notes 一覧取得

- `GET /api/notes`

クエリ例:

- `q`: 検索語
- `tag`: タグ名
- `sort`: `updated_desc` / `updated_asc` / `created_desc`
- `archived`: `true` / `false`

レスポンス例:

```json
{
  "items": [
    {
      "id": "uuid",
      "title": "daily note",
      "excerpt": "today i learned...",
      "tags": ["daily", "idea"],
      "createdAt": "2026-03-21T10:00:00Z",
      "updatedAt": "2026-03-21T12:00:00Z"
    }
  ]
}
```

#### note 詳細取得

- `GET /api/notes/:id`

レスポンス例:

```json
{
  "id": "uuid",
  "title": "daily note",
  "contentMd": "# hello",
  "tags": ["daily", "idea"],
  "createdAt": "2026-03-21T10:00:00Z",
  "updatedAt": "2026-03-21T12:00:00Z"
}
```

#### note 作成

- `POST /api/notes`

リクエスト例:

```json
{
  "title": "new memo",
  "contentMd": "",
  "tags": ["inbox"]
}
```

#### note 更新

- `PUT /api/notes/:id`

リクエスト例:

```json
{
  "title": "updated title",
  "contentMd": "# updated",
  "tags": ["work"]
}
```

#### note 削除

- `DELETE /api/notes/:id`

#### tags 一覧取得

- `GET /api/tags`

レスポンス例:

```json
{
  "items": [
    { "id": "uuid", "name": "daily" },
    { "id": "uuid", "name": "work" }
  ]
}
```

### 10.2 API 設計の原則

- UI に必要な形でレスポンスを返す
- DB スキーマをそのまま露出しすぎない
- ローカル専用でも入力バリデーションを行う
- タグは name ベースで扱ってもよいが、内部では id を持つ
- フロントエンドは API 経由でのみデータに触れ、Node.js 実装に直接依存しない
- 将来 Go に移行しても API 契約を維持できるようにする

## 11. フロントエンド設計

### 11.1 主なコンポーネント

- `AppLayout`
- `Sidebar`
- `SearchBar`
- `NoteList`
- `NoteListItem`
- `NoteEditor`
- `RichTextEditor`
- `TagPicker`
- `NoteMeta`

### 11.2 状態の分け方

- URL またはルーター state:
  - 選択中のメモ ID
  - 検索条件
  - 並び替え条件
- サーバー同期 state:
  - メモ一覧
  - メモ詳細
  - タグ一覧
- ローカル UI state:
  - 編集中テキスト
  - プレビュー表示有無
  - 保存中フラグ

### 11.3 編集体験

- エディタは `Tiptap` を採用する
- 表示状態と編集状態をできるだけ自然につなげる
- Markdown 記法に近い入力感を残しつつ、見た目はリッチに表示する
- Enter や記法の反映により、その場で見た目が整う体験を重視する
- 保存は自動保存を基本とする
- 入力停止後に短い debounce をかけて保存してもよい
- `Cmd/Ctrl + S` にも対応すると安心

### 11.4 エディタ設計の補足

- Notion ライクな体験は絶対条件とする
- そのため、単純な `textarea + preview` 構成は採用しない
- 初期段階では、Tiptap 上で必要最小限のブロック表現を実装する
- 完全な Notion 再現ではなく、個人用メモとして気持ちよく書ける体験を優先する

## 12. バックエンド設計

### 12.1 責務

- ローカル HTTP サーバー
- API ルーティング
- SQLite 接続
- マイグレーション
- 静的ファイル配信
- ブラウザ起動
- フロントエンドと疎結合な API の維持

### 12.2 セキュリティ方針

- `127.0.0.1` のみに bind
- 外部ネットワークへ公開しない
- CORS は必要最低限
- ファイルアップロード機能を入れる場合は保存先と拡張子を制限する
- Markdown 表示時は HTML サニタイズを行う

### 12.3 エラーハンドリング

- API エラーは JSON で返す
- ユーザー向けには簡潔な文言を表示
- DB ロックや保存失敗時は再試行しやすい設計にする

## 13. 起動と配布方針

### 13.1 目標

- GitHub から取得しやすい
- なるべくセットアップを少なくする
- ダブルクリックに近い体験で起動する

### 13.2 配布イメージ

#### 開発者向け

- ソースコードを clone
- `frontend` と `backend` を起動

#### 利用者向け

- GitHub Releases から配布物をダウンロード
- 起動ファイルを実行
- ブラウザが自動で開く
- 初期目標として Windows の社用 PC で起動できる状態を目指す

現状の実装ベースでは、Node.js 環境があることを前提に `node scripts/build-app.mjs` と `node scripts/start-app.mjs` で本番相当起動する。
あわせて、OS 別の薄いダブルクリックランチャーからこれらを呼び出す方針とする。

### 13.3 ランチャーの責務

- 既に起動中か確認する
- 未起動ならサーバーを起動する
- 空きポートを選ぶ
- ブラウザで URL を開く

### 13.4 補足

本プロジェクトでは `Google Chrome` を起動対象として固定する。  
Chrome が見つからない場合は、別ブラウザへフォールバックせず起動失敗として扱う。

## 15. 将来拡張

### 15.1 配布・展開の将来候補

今後の配布方式は、次の 3 案を比較して判断する。

#### A. `.app` / `.exe` 化

- 最も利用者体験がよい
- Node / npm やブラウザ起動方式を隠蔽しやすい
- Electron などを前提に、起動・更新・配布方式の再設計が必要

#### B. Node 同梱版

- 現在の `React + Node.js + SQLite` 構成を活かしやすい
- Node 未導入環境への配布障壁を下げやすい
- `.app` / `.exe` ほど自然な配布体験ではないが、実装コストは比較的抑えやすい

#### C. Web サービス化

- 配布不要で複数端末からアクセスしやすい
- ただし認証、認可、秘密管理、BYOK、安全性、運用保守などの設計負荷が大きい
- `memo4me` のローカル完結性とは方向性が変わる

現時点では、ローカル完結と BYOK の扱いやすさを活かせる `ローカルアプリ寄り` の方向を優先する。

### 15.2 機能面の将来拡張

- 画像添付
- アーカイブ
- お気に入り
- カスタムプロパティ
- SQLite FTS による全文検索強化
- 保存先ディレクトリの切り替え
- Git やクラウドストレージ経由のゆるい同期
- 家庭内 LAN 共有
- 認証付きの家庭内公開

## 16. 最終的な推奨構成

現時点の要件に対する推奨構成は以下。

- フロントエンド: `React + Vite + TypeScript`
- エディタ: `Tiptap`
- バックエンド: 初期は `Node.js`、将来 `Go` 移行を検討
- DB: `SQLite`
- 配布: 現在は `GitHub Releases + Node 非同梱 zip`
